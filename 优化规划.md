# TimeFloating 优化规划

> 面向当前代码现状整理的优化执行文档，用于指导下一阶段重构和功能扩展。

---

## 1. 当前状态判断

项目已经具备可用的第一版能力：

- 支持主面板与悬浮窗双窗口模式
- 已完成时钟、秒表、倒计时三种核心功能
- 已具备基础个性化设置、透明悬浮、置顶、锁定穿透等桌面体验

但从工程角度看，当前更偏向“功能完成版”，还不是“易维护、易扩展、易发布”的稳定产品版。

主要原因：

- 视图文件体量过大，职责耦合严重
- 状态同步逻辑分散在多个层级
- 配置存储简单但扩展性不足
- 自动化测试覆盖不够
- 跨平台适配尚未形成明确边界

---

## 2. 优化目标

下一阶段建议围绕以下四个目标推进：

### 2.1 提升可维护性
- 降低单文件复杂度
- 明确主进程、主面板、悬浮窗的职责边界
- 减少重复逻辑

### 2.2 提升可扩展性
- 为后续加入托盘、通知、快捷键、番茄统计等功能预留结构
- 让配置模型、状态模型、IPC 模型更稳定

### 2.3 提升桌面产品完整度
- 补足桌面应用高频能力，如托盘、通知、启动恢复、位置记忆

### 2.4 提升发布准备度
- 为 Windows 适配、后续打包和回归测试打基础

---

## 3. 优先优化项

### P0：先处理结构问题

#### 3.1 拆分主面板逻辑

当前问题：

- `src/views/MainPanel.vue` 同时负责展示、配置、计时、悬浮控制、IPC 同步
- 后续继续在这个文件上加功能，修改风险会快速上升

建议拆分：

- `src/components/clock/`
- `src/components/stopwatch/`
- `src/components/countdown/`
- `src/components/floating/`
- `src/composables/useClock.js`
- `src/composables/useStopwatch.js`
- `src/composables/useCountdown.js`
- `src/composables/useFloatingSync.js`

#### 3.2 拆分悬浮窗逻辑

当前问题：

- 悬浮窗本身承担了显示、按钮交互、计时校正、窗口适配等多项职责
- 与主面板存在重复的时间推导逻辑

建议方向：

- 保留悬浮窗作为轻量展示层
- 共享格式化与状态推导逻辑
- 把窗口自适配、控制条显隐、锁定交互拆成独立模块

#### 3.3 统一状态模型

建议抽出统一状态结构，例如：

- `mode`
- `settings`
- `stopwatch`
- `countdown`
- `windowState`

这样后续扩展计次、番茄流程、统计数据时，不需要继续在多个页面里临时拼字段。

---

## 4. 基础能力优化项

### P1：配置存储升级

当前问题：

- 手写 JSON 文件读写，后续缺少版本迁移能力
- 自定义背景图直接写入配置内容，不利于长期使用

建议方案：

- 接入 `electron-store`
- 增加默认配置 schema
- 增加配置版本字段
- 图片迁移为文件路径引用

### P1：IPC 通信整理

当前问题：

- IPC channel 较多，维护上依赖人工记忆
- 缺少统一常量与明确的事件分类

建议方案：

- 新建 `src/constants/ipc.js` 或 `electron/ipc/channels.js`
- 区分：
  - 窗口控制类
  - 状态同步类
  - 配置读写类
  - 用户操作类

### P1：测试基础补齐

当前已有：

- `test/time.test.js`

建议补充：

- 秒表开始 / 暂停 / 重置的状态测试
- 倒计时剩余时间推导测试
- 配置默认值与配置迁移测试
- 悬浮窗同步 payload 测试

这些功能与项目定位高度匹配，优先级较高：

1. 托盘 / 菜单栏模式

3. 开机自启动
4. 悬浮窗位置记忆

2. 番茄钟流程预设

---

## 6. 建议实施顺序

建议分 4 个阶段推进，不要同时大范围改功能和结构。

### 阶段一：结构整理

目标：

- 降低文件复杂度
- 提炼可复用逻辑

任务：

1. 拆分时间逻辑为 composables
2. 抽出统一状态与格式化层
3. 收敛主面板与悬浮窗的重复逻辑

### 阶段二：基础设施升级

目标：

- 提升稳定性

任务：

1. 配置迁移到 `electron-store`
2. 统一 IPC 常量与数据结构
3. 补基础测试

### 阶段三：桌面能力增强

目标：

- 提升产品完整度

任务：

1. 增加通知
2. 增加托盘 / 菜单栏
3. 增加窗口位置恢复与启动恢复

### 阶段四：功能扩展

目标：

- 提升差异化与使用价值

任务：

1. 秒表计次
2. 番茄钟流程
3. 统计能力
4. 平台扩展

---

## 7. 当前最适合先做的事项

如果马上开始优化，建议第一批只做下面几项：

1. 抽离 `time`、`countdown`、`floating sync` 逻辑
2. 接入 `electron-store`
3. 增加通知能力
4. 增加托盘 / 菜单栏模式

原因：

- 这几项既能明显改善代码结构，也能明显提升产品完成度
- 不会一开始就把优化范围拉得过大
- 做完后再继续扩展番茄钟、统计或 Windows 适配会更稳

---

## 8. 风险提醒

### 8.1 不建议直接继续在现有大文件上叠功能

如果继续在 `MainPanel.vue` 和 `FloatingWindow.vue` 上直接追加托盘、通知、计次、统计，短期能做出来，但后续维护成本会明显失控。

### 8.2 不建议一开始就同时做跨平台

当前项目主力适配 macOS 是合理的。应先把结构、配置、桌面基础能力整理好，再做 Windows 兼容，不然调试面会过大。

### 8.3 不建议把背景图片继续直接写进配置

这会导致配置文件持续增大，也会让后续同步、迁移和调试变得更麻烦。

---

## 9. 结论

当前项目基础不错，方向也成立，适合继续打磨。

下一步最合理的策略不是直接堆更多功能，而是：

1. 先整理结构
2. 再补桌面应用基础能力
3. 最后扩展效率功能

这样能保证后续优化是真正“越做越稳”，而不是“功能越多越难改”。
