## 二、核心知识点详解

### 2.1 Electron 双窗口架构

**问题**：为什么需要双窗口？主面板和悬浮窗的关系是什么？

**回答要点**：
- **主面板**（`#/`）：完整的控制与配置界面，左侧大字展示当前时间，右侧提供功能切换与个性化设置
- **悬浮窗**（`#/floating`）：无边框透明胶囊小窗，可自由拖拽，支持锁定穿透模式
- **关系**：主面板是计时状态的唯一权威来源，悬浮窗是"只读展示层 + 操作上报层"

**架构图**：
```
主进程 (electron/main.js)
├── mainWindow     →  加载 /#/          (主面板)
└── floatingWindow →  加载 /#/floating  (悬浮窗)

IPC 通信流程：
主面板 --[状态快照]--> 主进程缓存 --[转发]--> 悬浮窗
悬浮窗 --[操作指令]--> 主进程 --[转发]--> 主面板
```

### 2.2 IPC 通信机制设计

**问题**：主面板和悬浮窗是如何通信的？为什么这样设计？

**回答要点**：

1. **三层架构**：渲染进程 ↔ 主进程 ↔ 渲染进程
2. **状态同步策略**：
   - **低频快照同步**（`pushFloatingState`）：在关键动作（开始/暂停/重置）后发送完整状态
   - **高频 tick 推送**（`pushTimerTick`）：每 80ms 发送实时数据，只包含变化字段
3. **为什么不用 SharedWorker/MessageChannel**：
   - Electron 的渲染进程是隔离的，不能直接通信
   - 主进程作为中转可以做缓存、校验、权限控制

**代码示例**（主进程）：
```javascript
// 主面板推送计时器实时数据 → 转发给悬浮窗
ipcMain.on("push-timer-tick", (_event, data) => {
  // tick 不入缓存，只做瞬时转发
  if (floatingWindow && floatingWindow.webContents) {
    floatingWindow.webContents.send("timer-tick", data);
  }
});

// 悬浮窗操作 → 转发给主面板
ipcMain.on("floating-action", (_event, action) => {
  // 悬浮窗不直接改主进程缓存，只上报动作给主面板处理
  if (mainWindow && mainWindow.webContents) {
    mainWindow.webContents.send("floating-action", action);
  }
});
```

### 2.3 Preload + ContextBridge 安全通信

**问题**：为什么使用 preload？直接用 ipcRenderer 不行吗？

**回答要点**：

1. **安全考虑**：`contextIsolation: true` + `nodeIntegration: false`
   - 渲染进程不能直接访问 Node.js API
   - 防止恶意脚本调用系统能力
2. **白名单模式**：只暴露需要的 API，不暴露整个 ipcRenderer
3. **封装原则**：渲染进程不知道 IPC 细节，只调用业务语义化的方法

**代码示例**（preload.js）：
```javascript
contextBridge.exposeInMainWorld("electronAPI", {
  // 窗口控制
  openFloating: (floatingState) => ipcRenderer.send("open-floating", floatingState),
  closeFloating: () => ipcRenderer.send("close-floating"),
  
  // 数据持久化
  saveSettings: (settings) => ipcRenderer.invoke("save-settings", settings),
  getSettings: () => ipcRenderer.invoke("get-settings"),
  
  // 状态同步
  pushFloatingState: (state) => ipcRenderer.send("push-floating-state", state),
  getFloatingState: () => ipcRenderer.invoke("get-floating-state"),
  
  // 监听器（带清理）
  onStateUpdate: (callback) => {
    ipcRenderer.on("state-update", (_event, state) => callback(state));
  },
  removeStateUpdateListener: () => ipcRenderer.removeAllListeners("state-update"),
});
```

### 2.4 RAF 驱动的精确计时

**问题**：为什么用 requestAnimationFrame 而不是 setInterval？如何保证毫秒精度？

**回答要点**：

1. **RAF 优势**：
   - 与浏览器刷新率同步（通常 60fps，16.67ms 一帧）
   - 页面不可见时自动暂停，节省资源
   - 比 setInterval 更平滑，不会出现"跳帧"
2. **时间戳推导**：
   - 不用"累加"方式（`time += 16`），而是用 `Date.now() - startTime`
   - 避免 setInterval 误差累积
3. **结束时间戳**：
   - 倒计时记录 `endAt = Date.now() + totalMs`
   - 每帧计算 `remaining = endAt - Date.now()`
   - 避免"首秒多算"问题

**代码示例**：
```javascript
// RAF 驱动循环
startRaf() {
  const loop = () => {
    const now = Date.now();
    // 驱动 clockMs / countdownMs 等依赖 _msTick 的 computed
    this._msTick = now;
    
    // 秒表：运行中每帧更新，保证毫秒精确
    if (this.stopwatchRunning && this._swStart !== undefined) {
      this.stopwatchTime = now - this._swStart;
    }
    
    // 倒计时：按结束时间戳推导剩余时间
    if (this.countdownRunning && this._cdEndAt !== undefined) {
      const remainingMs = Math.max(0, this._cdEndAt - now);
      this.syncCountdownFromRemainingMs(remainingMs);
      if (remainingMs <= 0) this.finishCountdown();
    }
    
    this._rafHandle = requestAnimationFrame(loop);
  };
  this._rafHandle = requestAnimationFrame(loop);
}
```

### 2.5 悬浮窗拖拽实现

**问题**：为什么不用 `-webkit-app-region: drag`？JS 拖拽怎么实现？

**回答要点**：

1. **问题**：`-webkit-app-region: drag` 会吞掉鼠标事件，导致按钮点击失效
2. **解决方案**：主进程轮询鼠标坐标驱动
3. **实现原理**：
   - 渲染进程发送 `start-drag`
   - 主进程记录初始位置，启动 16ms 定时器轮询光标坐标
   - 计算偏移量，调用 `setBounds` 移动窗口

**代码示例**（主进程）：
```javascript
ipcMain.on("start-drag", () => {
  const { screen } = require("electron");
  const cursor = screen.getCursorScreenPoint();
  const bounds = floatingWindow.getBounds();
  dragStartPos = cursor;
  dragStartBounds = bounds;

  // 用主进程轮询鼠标位置来驱动拖拽
  dragInterval = setInterval(() => {
    const cur = screen.getCursorScreenPoint();
    const dx = cur.x - dragStartPos.x;
    const dy = cur.y - dragStartPos.y;
    floatingWindow.setBounds({
      x: dragStartBounds.x + dx,
      y: dragStartBounds.y + dy,
      width: dragStartBounds.width,
      height: dragStartBounds.height,
    });
  }, 16); // 约 60fps
});
```

### 2.6 鼠标穿透锁定

**问题**：什么是鼠标穿透？如何实现 Hover 解锁？

**回答要点**：

1. **应用场景**：锁定模式下，悬浮窗不阻挡鼠标操作
2. **API**：`setIgnoreMouseEvents(ignore, { forward: true })`
   - `ignore: true`：鼠标事件穿透窗口
   - `forward: true`：仍然能检测到鼠标进入/离开
3. **Hover 解锁**：
   - 渲染进程监听 `mousemove`
   - 判断鼠标是否在锁定按钮区域
   - 动态切换穿透状态

**代码示例**（渲染进程）：
```javascript
onRootMouseMove(e) {
  if (this.locked) {
    // 锁定模式下：鼠标在锁定按钮上时恢复可点击
    const btn = this.$refs.lockBtn;
    if (btn) {
      const rect = btn.getBoundingClientRect();
      const over = e.clientX >= rect.left && e.clientX <= rect.right &&
                   e.clientY >= rect.top && e.clientY <= rect.bottom;
      window.electronAPI.setIgnoreMouseEvents(!over);
    }
    return;
  }
  this.revealControls();
}
```

### 2.7 自适应窗口宽度

**问题**：悬浮窗如何根据内容自动调整大小？

**回答要点**：

1. **胶囊元素**：无固定宽度，由内容撑开
2. **测量时机**：字体变化、毫秒开关、模式切换时触发
3. **测量方法**：`scrollWidth` 反映实际内容宽度
4. **同步窗口**：调用 `resizeFloating(w + 20, h + 20)`，留出阴影边距

**代码示例**：
```javascript
fitWindow() {
  const el = this.$refs.capsule;
  if (!el || !window.electronAPI) return;
  // scrollWidth 能反映实际内容宽度
  const w = Math.max(220, el.scrollWidth);
  const h = el.offsetHeight;
  // 窗口 = 胶囊尺寸 + 四周留出 20px 透明边距
  window.electronAPI.resizeFloating(w + 20, h + 20);
}
```

### 2.8 配置持久化

**问题**：如何保存用户设置？为什么用 JSON 文件而不是 localStorage？

**回答要点**：

1. **存储位置**：`~/.floating-clock-config.json`
2. **为什么不用 localStorage**：
   - Electron 渲染进程的 localStorage 是隔离的
   - 两个窗口的 localStorage 不共享
   - 主进程无法直接访问 localStorage
3. **实现方式**：主进程通过 `fs` 模块读写 JSON 文件

**代码示例**（主进程）：
```javascript
const configPath = path.join(os.homedir(), ".floating-clock-config.json");

function saveConfig(data) {
  try {
    fs.writeFileSync(configPath, JSON.stringify(data, null, 2));
    return true;
  } catch (error) {
    console.error("保存配置失败:", error);
    return false;
  }
}

function loadConfig() {
  try {
    if (fs.existsSync(configPath)) {
      return JSON.parse(fs.readFileSync(configPath, "utf8"));
    }
  } catch (error) {
    console.error("读取配置失败:", error);
  }
  return { /* 默认配置 */ };
}
```

## 三、项目难点与解决方案

### 3.1 难点一：双窗口状态同步

**问题描述**：
- 主面板和悬浮窗是两个独立的 Vue 应用
- 需要保持计时状态一致
- 高频更新（毫秒级）不能阻塞 UI

**解决方案**：
1. **主面板是权威来源**：所有计时逻辑在主面板执行
2. **悬浮窗本地乐观更新**：点击按钮后立即更新 UI，不等待 IPC 响应
3. **状态校正机制**：主面板定期推送快照，悬浮窗用快照校正本地状态

**代码体现**：
```javascript
// 悬浮窗：乐观更新
handleAction(action) {
  this.applyOptimisticAction(action); // 立即更新本地
  window.electronAPI.floatingAction(action); // 通知主面板
}

// 主面板：监听操作并推送快照
window.electronAPI.onFloatingAction((action) => {
  if (action === "stopwatch-toggle") this.toggleStopwatch();
  // ... 执行完后推送快照
  this.pushFloatingSnapshot();
});
```

### 3.2 难点二：倒计时首秒多算问题

**问题描述**：
- 使用 `setInterval(1000)` 倒计时，第一秒会显示完整时间
- 用户点击开始后，应该立即进入下一秒段

**解决方案**：
1. **记录结束时间戳**：`endAt = Date.now() + totalMs`
2. **按剩余时间推导**：`remaining = endAt - Date.now()`
3. **开区间处理**：`normalizedMs = Math.max(0, remainingMs - 1)`

**代码体现**：
```javascript
toggleCountdown() {
  const total = this.countdownHours * 3600 + this.countdownMinutes * 60 + this.countdownSeconds;
  if (total <= 0) return;
  
  // 记录结束时间戳
  this._cdEndAt = Date.now() + total * 1000;
  this.syncCountdownFromRemainingMs(total * 1000);
}

syncCountdownFromRemainingMs(remainingMs) {
  // 倒计时展示按"剩余开区间"处理
  const normalizedMs = Math.max(0, remainingMs - 1);
  const totalSeconds = Math.floor(normalizedMs / 1000);
  
  this.countdownHours = Math.floor(totalSeconds / 3600);
  this.countdownMinutes = Math.floor((totalSeconds % 3600) / 60);
  this.countdownSeconds = totalSeconds % 60;
  this.countdownMs = normalizedMs % 1000;
}
```

### 3.3 难点三：透明窗口拖拽与鼠标事件冲突

**问题描述**：
- 透明窗口需要拖拽，但 `-webkit-app-region: drag` 会吞掉鼠标事件
- 按钮点击、Hover 检测都会失效

**解决方案**：
1. **主进程轮询**：用 `setInterval(16)` 轮询光标坐标
2. **渲染进程触发**：只在拖拽区域触发 `start-drag`
3. **事件分离**：拖拽和点击使用不同的事件处理

**代码体现**：
```javascript
// 渲染进程
onDragStart(e) {
  // 如果点击的是按钮，不启动拖拽
  if (e.target.closest("button")) return;
  window.electronAPI.startDrag();
  
  const onUp = () => {
    window.electronAPI.stopDrag();
    document.removeEventListener("mouseup", onUp);
  };
  document.addEventListener("mouseup", onUp);
}
```

### 3.4 难点四：跨窗口 IPC 通信设计

**问题描述**：
- 两个渲染进程不能直接通信
- 需要设计高效、可靠的通信协议

**解决方案**：
1. **分层设计**：
   - 低频快照（`pushFloatingState`）：完整状态，用于初始化和校正
   - 高频 tick（`pushTimerTick`）：增量数据，用于实时显示
2. **缓存机制**：主进程缓存最新状态，悬浮窗启动时直接读取
3. **清理机制**：提供 `removeXxxListener` 方法，防止内存泄漏

## 四、高频面试题与参考答案

### Q1：介绍一下你的项目

**参考答案**：
这是一个基于 Vue 3 + Electron 的 macOS 桌面悬浮时钟应用。项目采用双窗口架构，主面板负责配置和控制，悬浮窗负责展示和快捷操作。两者通过主进程 IPC 通信，实现状态同步。

技术上，我实现了几个亮点：
1. 使用 requestAnimationFrame 驱动秒表和倒计时，保证毫秒级精度
2. 设计了状态快照 + 高频 tick 的分层同步机制
3. 实现了主进程轮询的窗口拖拽，解决了透明窗口鼠标事件冲突
4. 通过 preload + contextBridge 封装白名单 API，保证安全性

### Q2：项目中遇到的最大挑战是什么？

**参考答案**：
最大的挑战是双窗口的状态同步问题。

最初我尝试用 SharedWorker，但发现 Electron 的渲染进程是隔离的，不能直接通信。后来设计了主进程中转的 IPC 机制，但又遇到高频更新（毫秒级）导致 IPC 消息过多的问题。

最终的解决方案是分层设计：
- 低频快照：在关键动作（开始/暂停/重置）后发送完整状态
- 高频 tick：每 80ms 发送一次，只包含变化字段
- 悬浮窗本地乐观更新：点击按钮后立即更新 UI，不等待 IPC 响应

这样既保证了状态一致性，又不会阻塞 UI 渲染。

### Q3：为什么用 requestAnimationFrame 而不是 setInterval？

**参考答案**：
主要有三个原因：

1. **精度**：RAF 与浏览器刷新率同步（60fps，16.67ms 一帧），比 setInterval 更平滑
2. **性能**：页面不可见时 RAF 自动暂停，节省资源
3. **避免误差累积**：我用时间戳推导（`Date.now() - startTime`）而不是累加（`time += 16`），避免 setInterval 的误差累积

对于倒计时，我记录结束时间戳（`endAt = Date.now() + totalMs`），每帧计算剩余时间（`remaining = endAt - Date.now()`），这样即使页面切到后台再切回来，时间也是准确的。

### Q4：如何实现鼠标穿透锁定？

**参考答案**：
使用 Electron 的 `setIgnoreMouseEvents(ignore, { forward: true })` API。

- `ignore: true`：鼠标事件穿透窗口，不会阻挡下层窗口的操作
- `forward: true`：仍然能检测到鼠标进入/离开事件

实现 Hover 解锁的逻辑是：
1. 渲染进程监听 `mousemove` 事件
2. 判断鼠标是否在锁定按钮区域（通过 `getBoundingClientRect`）
3. 动态切换穿透状态：在按钮上时恢复可点击，其他区域穿透

### Q5：为什么用 JSON 文件存储配置而不是 localStorage？

**参考答案**：
主要有三个原因：

1. **窗口隔离**：Electron 的每个渲染进程有独立的 localStorage，两个窗口的配置不共享
2. **主进程访问**：主进程无法直接访问渲染进程的 localStorage
3. **持久化**：JSON 文件存储在用户目录（`~/.floating-clock-config.json`），应用卸载重装后配置不丢失

实现上，主进程通过 `fs` 模块读写 JSON 文件，渲染进程通过 IPC 调用 `saveSettings` 和 `getSettings`。

### Q6：如何保证悬浮窗和主面板的状态一致性？

**参考答案**：
采用"权威来源 + 乐观更新 + 状态校正"的三层机制：

1. **权威来源**：主面板是计时状态的唯一权威，所有业务逻辑在主面板执行
2. **乐观更新**：悬浮窗点击按钮后立即更新本地 UI，不等待 IPC 响应，提升用户体验
3. **状态校正**：主面板在关键动作后推送完整快照，悬浮窗用快照校正本地状态

这样即使 IPC 有延迟，用户操作也能立即响应，同时保证最终状态一致。

### Q7：IPC 通信是如何设计的？

**参考答案**：
采用分层设计：

1. **低频快照**（`pushFloatingState`）：
   - 触发时机：开启悬浮窗、关键动作后
   - 内容：完整状态（模式、设置、计时数据）
   - 用途：初始化和状态校正

2. **高频 tick**（`pushTimerTick`）：
   - 触发时机：每 80ms
   - 内容：只包含变化字段（秒表时间、倒计时秒数）
   - 用途：实时显示更新

3. **操作指令**（`floatingAction`）：
   - 触发时机：用户点击按钮
   - 内容：动作标识（`stopwatch-toggle`、`countdown-reset`）
   - 用途：悬浮窗操作回传给主面板

4. **缓存机制**：主进程缓存最新状态，悬浮窗启动时直接读取（`getFloatingState`）

### Q8：如何处理窗口拖拽和按钮点击的冲突？

**参考答案**：
最初尝试用 `-webkit-app-region: drag`，但发现它会吞掉鼠标事件，导致按钮点击失效。

最终采用主进程轮询的方案：
1. 渲染进程在拖拽区域发送 `start-drag`
2. 主进程记录初始位置，启动 16ms 定时器轮询光标坐标
3. 计算偏移量，调用 `setBounds` 移动窗口
4. 松开鼠标时发送 `stop-drag`，清除定时器

关键点是判断点击目标：如果点击的是按钮（`e.target.closest("button")`），不启动拖拽。

## 五、技术深度拓展

### 5.1 Electron 安全最佳实践

```javascript
webPreferences: {
  preload: path.join(__dirname, "preload.js"),
  contextIsolation: true,  // 渲染进程与 preload 脚本隔离
  nodeIntegration: false,  // 禁止渲染进程访问 Node.js API
}
```

- **contextIsolation**：防止渲染进程篡改 preload 暴露的 API
- **nodeIntegration**：关闭 Node.js 直连，避免安全风险
- **白名单模式**：只暴露业务需要的 API，不暴露整个 ipcRenderer

### 5.2 Vue 3 响应式优化

```javascript
// 使用 void 建立响应式依赖
clockMs() {
  void this._msTick; // 建立依赖，RAF 更新时触发重新计算
  return Math.floor((Date.now() % 1000) / 100);
}
```

- `_msTick` 由 RAF 每帧更新
- `void` 操作建立响应式依赖
- computed 在 `_msTick` 变化时重新计算

### 5.3 窗口生命周期管理

```javascript
// 悬浮窗关闭时通知主面板
floatingWindow.on("closed", () => {
  floatingWindow = null;
  notifyFloatingClosed();
});

function notifyFloatingClosed() {
  if (mainWindow && mainWindow.webContents) {
    mainWindow.webContents.send("floating-action", "close");
    mainWindow.show(); // 恢复主面板显示
  }
}
```

### 5.4 性能优化策略

1. **RAF 节流**：只在运行中更新，暂停时不消耗资源
2. **增量同步**：高频 tick 只发送变化字段，减少 IPC 数据量
3. **延迟隐藏**：Hover 控件延迟 180ms 隐藏，避免鼠标移动时闪烁
4. **清理监听器**：`beforeUnmount` 时移除所有 IPC 监听，防止内存泄漏
